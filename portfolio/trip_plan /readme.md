# Trip Plan - Мобильное приложение для планирования путешествий

## О проекте

Trip Plan - мобильное приложение-спутник путешественника, разрабатываемое для туристического агентства. Приложение решает проблему «информационного хаоса»: объединяет в одном месте документы, маршруты, расходы и точки интереса, обеспечивая офлайн-доступ в поездке.

**Роль в проекте:** системный аналитик - от сбора требований через интервью с заказчиком до проектирования REST API и структуры базы данных.

### Контекст

- **Заказчик:** туристическое агентство (посредник между туроператорами и клиентами)
- **Платформы:** iOS 14+, Android 9+ (кроссплатформенная разработка)
- **Пользователи:** путешественники (28-50 лет) и менеджеры агентства
- **Масштаб MVP:** 5 000 → 100 000 MAU за первый год

## Артефакты

### 1. Бриф на создание приложения [Trip plan](docs/brief.md)

Структурированные результаты интервью с заказчиком: бизнес-цели, целевая аудитория, пользовательские роли, сценарии монетизации, требования к офлайн-режиму, интеграции и ограничения MVP. Документ содержит 35 вопросов с ответами по всем аспектам будущей системы.

### 2. [Формализованные требования](docs/requirements.md)

Требования к системе, формализованные на основе брифа. Каждое требование имеет уникальный ID, источник, приоритет (MoSCoW) и тип.

- **Функциональные требования:** ФТ01-ФТ09 (пользовательские)
- **Приоритизация:** Must have / Should have / Could have

### 3. Варианты использования [(Use Cases)](docs/use_cases.md)

Диаграмма вариантов использования [(UML Use Case)](diagrams/use-case-diagram.png) и текстовые описания 6 основных сценариев:

| ID | Вариант использования |
|----|-----------------------|
| UC-01 | Зарегистрироваться / войти в приложение |
| UC-02 | Создать и управлять поездкой |
| UC-03 | Управлять документами и расходами |
| UC-04 | Работать с точками интереса (POI) |
| UC-05 | Управлять совместной поездкой |
| UC-06 | Приобрести дополнительную услугу |

Каждый Use Case содержит: акторов, триггер, предусловия, основной и альтернативные потоки, исключения.

### 4. Спецификация требований [(SRS)](docs/srs.md)

Основной документ проекта, написанный по структуре IEEE 830. Включает:

- Видение продукта и перспективы развития (этапы 1-3)
- Классы пользователей: неавторизованный, клиент, администратор
- **30 функциональных требований** (ФТ01-ФТ30) по 8 блокам: аутентификация, управление поездками, документы/расходы/заметки, POI, совместные поездки, доп. услуги, уведомления, администрирование
- **9 нефункциональных требований** (НФТ01-НФТ09): производительность, безопасность, атрибуты качества
- Внешние интеграции: SMS-шлюз, push-сервис, платёжный агрегатор, OAuth (VK/Telegram/Яндекс), S3-хранилище

### 5. [ER-диаграмма](diagrams/er-diagram.png)

Диаграмма "сущность-связь", описывающая структуру базы данных: пользователи, поездки, участники, документы, расходы, заметки, POI, дополнительные услуги, уведомления и справочники.

### 6. Спецификация [REST API](api/openapi.yaml)

Полная спецификация программного интерфейса в формате OpenAPI 3.0.3:

- **45 путей (paths)**, **72 эндпоинта**, **68 схем данных**
- 15 тегов: регистрация, пользователи, поездки, участники, документы, расходы, POI, заметки, каталог POI, доп. услуги, покупки, поддержка, уведомления, отчёты, справочники
