## 1. Введение

### 1.1. Назначение

Настоящая спецификация требований к программному обеспечению (SRS) описывает функциональные и нефункциональные требования к выпуску 1.0 (MVP) мобильного приложения Trip plan.

Спецификация предназначена для команд разработки, тестирования, аналитики, а также для заказчика (туристическое агентство) в качестве основы для приёмки результатов работ.

Кроме специально обозначенных случаев, все указанные здесь требования имеют высокий приоритет и приписаны к выпуску 1.0.


### 1.2. Область применения

Trip plan - мобильное приложение-спутник путешественника, разрабатываемое для туристического агентства и решающее ключевые боли:
-	устранение “информационного хаоса” (документы/обсуждения в почте и чатах);
-	доступ к документам и планам в поездке (включая офлайн);
-	возможность агентству предлагать дополнительные услуги в нужный момент;
-	снижение нагрузки на менеджеров по рутинным вопросам.
Полная функциональность перечислена в разделе 3. “Функциональные требования”.


### 1.3. Определения, акронимы и сокращения

1.	**MVP** - минимально жизнеспособная версия продукта.
2.	**OAuth** – протокол авторизации через внешних поставщиков идентификации (VK, Telegram, Яндекс).
3.	**OTP** - одноразовый код подтверждения (SMS).
4.	**POI (Point of Interest)** - точка интереса (место/объект) из каталога.
5.	**Авторизация** – это процесс предоставления определенному пользователю (или группе) прав доступа к ресурсам, функциям или данным системы, а также проверка этих прав при попытке выполнить действие.  
6.	**Администратор (менеджер)** - сотрудник туристического агентства с административными правами.
7.	**Дополнительная услуга** – платная услуга агентства (экскурсия, трансфер, страховка), доступная для приобретения через приложение.
8.	**Офлайн-режим** - доступ к данным без сети, с последующей синхронизацией.
9.	**Поездка (путешествие)** – центральная сущность приложения, описывающая путешествие с указанием страны, городов, дат и связанного контента.
10.	**ПО** – программное обеспечение.
11.	**Пользователь (клиент)** - конечный пользователь приложения, путешественник.
12.	**Приложение/Trip plan** - мобильное приложение (iOS/Android), разработанное в соответствии с данной спецификацией.
13.	**Синхронизация** – процесс передачи изменений, выполненных в офлайн-режиме, на сервер после восстановления сетевого подключения. 
14.	**Участник поездки** – пользователь, добавленный владельцем поездки в качестве соучастника с определенным уровнем доступа (владелец/редактор/наблюдатель).

## 2. Общее описание

### 2.1. Видение продукта

Приложение Trip plan представляет собой мобильный “цифровой спутник” путешественника, который объединяет в одном месте всю информацию о поездке. Приложение позволяет клиенту создавать и вести поездки, хранить и открывать документы, планировать маршрут с помощью POI и карты, фиксировать расходы и видеть статистику по ним, а также покупать дополнительные услуги туристического агентства. Приложение снижает “информационный хаос” за счёт централизованного хранения данных и офлайн-доступа. Trip plan помогает агентству сопровождать клиента в поездке и повышать продажи доп. услуг за счёт релевантных предложений и уведомлений.

### 2.2. Перспективы продукта

На втором этапе развития приложения Trip plan можно расширить функциональность за счёт автоматизаций и интеграций, таких как:
-	автоматическое построение маршрута между выбранными точками;
-	автоматический парсинг данных из писем с бронированиями;
-	совместные списки покупок/сборов для участников поездки;
-	голосование за точки маршрута внутри группы.
	
Для повышения полезности в путешествии целесообразно добавить интеграции с погодными сервисами и использование внешних картографических платформ (интеграции Яндекс.Карты/Google Maps API), а также интеграцию с CRM туристического агентства для сквозной работы с клиентом и предложениями. Дополнительно можно предусмотреть отдельные интеграции через API погоды для сценариев предупреждений и рекомендаций по погодным условиям.

На третьем этапе возможно подключение API онлайн-киосков билетов (например, для покупки билетов в музеи и другие культурные объекты) с отображением доступности, стоимости и оформлением покупки непосредственно в приложении, английская локализация интерфейса и разработка адаптивного интерфейса для планшетов.

### 2.3. Взаимодействие продукта с другими продуктами и компонентами

В рамках MVP приложение взаимодействует со следующими внешними системами:

| Внешняя система | Назначение | Протокол |
|---|---|---|
| SMS-шлюз (оператор связи) | Отправка OTP-кодов при регистрации/входе, SMS-уведомления (резервный канал) | REST API |
| Сервис push-уведомлений | Доставка push-уведомлений пользователям | REST API |
| Платёжный агрегатор | Обработка платежей при покупке дополнительных услуг | REST API |
| Поставщики OAuth (VK, Telegram, Яндекс) | Авторизация пользователей через социальные сети | OAuth 2.0 |
| Объектное хранилище (S3-совместимое) | Хранение загруженных документов поездок | S3 API |

### 2.4. Классы и характеристики пользователей

| Класс | Характеристика |
|---|---|
| **Неавторизованный пользователь** | Посетитель, не прошедший авторизацию. Доступны только экраны регистрации и входа. |
| **Авторизованный пользователь (Клиент/Путешественник)** | Основной пользователь приложения. Обладает правами:<br>-	Создавать/редактировать/просматривать поездки;<br>-	Просматривать поездки, созданные менеджером/администратором;<br>-	Управлять участниками поездки;<br>-	Добавлять/редактировать/удалять заметки и расходы;<br>-	Загружать/скачивать/удалять документы поездки;<br>-	Участвовать в совместных действиях внутри поездки как приглашенный участник;<br>-	Добавлять POI в план поездки, просматривать POI на карте;<br>-	Просматривать свод расходов по поездке/категориям;<br>-	Просматривать каталог доп. услуг и оформлять покупку/заявку;<br>-	Управлять профилем и настройками уведомлений;<br>-	Удалять аккаунт.|
| **Администратор (Менеджер)** | Пользователь с расширенными административными правами. Доступны все функции клиента, а также обладает правами:<br>-	Управлять пользователями (в рамках политики доступа);<br>-	Создавать/редактировать поездки для клиентов;<br>-	Управлять каталогом POI;<br>-	Управлять каталогом дополнительных услуг;<br>-	Просматривать административные отчёты и статистику;<br>-	Обрабатывать обращения пользователей (поддержка/модерация);<br>-	Инициировать уведомления/напоминания. |

### 2.5. Ограничения разработки и реализации

К ограничениям разработки и реализации относятся:
-	Кроссплатформенная разработка (React Native/Flutter);
-	Минимально поддерживаемые версии: iOS 14.0, Android 9.0;
-	Формат REST API: OpenAPI 3.0.3, JSON;
-	Максимальный размер загружаемого файла: 10 МБ. Суммарный объём документов на поездку: 50 МБ;
-	Общий лимит облачного хранилища: 1 ГБ на пользователя;
-	Поддерживаемые форматы документов: PDF, JPG, PNG;
-	Язык интерфейса MVP - русский;
-	Хранение данных: гибридное (облако + устройство).

## 3. Функциональные требования

### 3.1. Аутентификация, авторизация и управление аккаунтом

#### 3.1.1. Описание и приоритет

Блок обеспечивает регистрацию, вход и управление учётной записью пользователя. Приоритет: **Must have**.

#### 3.1.2. Причинно-следственные связи

Аутентификация является входной точкой для всех остальных функциональных блоков. Без успешной авторизации пользователь не получает доступа к функциям приложения.

#### 3.1.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ18 | Система должна поддерживать регистрацию и вход пользователя с использованием номера телефона и одноразового кода подтверждения (OTP). | Must have |
| ФТ20 | Система должна обеспечивать отправку одноразового кода подтверждения на номер телефона пользователя и проверку кода при входе. | Must have |
| ФТ22 | Система должна поддерживать регистрацию и вход пользователя через внешние поставщики идентификации, определённые заказчиком. | Must have |
| ФТ23 | Система должна поддерживать роли Пользователь и Администратор. | Must have |
| ФТ10 | Пользователь должен иметь возможность удалить аккаунт с удалением персональных данных. | Must have |

#### 3.1.4. Алгоритмы

**UC-01-US/Зарегистрироваться/войти в приложение (Основной поток)**

1.	Если пользователь выбирает регистрацию по номеру телефона, то управление переходит на следующий шаг;
2.	Система выводит экран для ввода номера телефона;
3.	Пользователь вводит номер телефона;
4.	Пользователь нажимает на кнопку “Далее”;
5.	Если телефон пользователя не был зарегистрирован ранее, то управление переходит на следующий шаг;
6.	Система отправляет пользователю одноразовый код для входа в систему на номер телефона;
7.	Система выводит экран с полем ввода кода из СМС;
8.	Если пользователь получил одноразовый код в течение 5 минут, то управление переходит на следующий шаг;
9.	Пользователь вводит код из СМС;
10.	Система верифицирует введенные данные;
11.	Если введенный и отправленный ранее код совпадают, то управление переходит на следующий шаг;
12.	Система выводит экран с полями ввода имени и фамилии пользователя;
13.	Пользователь вводит свои имя и фамилию в форму;
14.	Пользователь нажимает на кнопку “Далее”;
15.	Система выводит алерт о согласии на обработку персональных данных;
16.	Пользователь нажимает на кнопку “Согласен с условиями обработки”;
17.	Система создает учетную запись для пользователя;
18.	Система выводит главную страницу приложения.

**UC-06-US/Удалить аккаунт (Основной поток)**

1.	Система отображает алерт об удалении аккаунта и персональных данных;
2.	Система запрашивает подтверждение пользователя;
3.	Если пользователь подтверждает удаление аккаунта, нажимая на кнопку “Удалить аккаунт”, то управление переходит на следующий шаг;
4.	Если удаление выполняется в онлайн режиме, то управление переходит на следующий шаг;
5.	Система выполняет удаление учетной записи пользователя и связанных персональных данных;
6.	Система завершает пользовательскую сессию и отображает стартовый экран приложения.

**UC-03-ITG/Отправить одноразовый код по СМС (Основной поток)**

1.	Система инициирует отправку одноразового кода на номер телефона пользователя;
2.	Система генерирует одноразовый код и сохраняет его для последующей верификации;
3.	Система формирует параметры СМС и текст сообщения с кодом;
4.	Система отправляет запрос во внешнюю систему уведомлений;
5.	Внешняя система уведомлений принимает запрос и возвращает результат обработки;
6.	Если результат успешный, то управление переходит на следующий шаг;
7.	Система фиксирует факт отправки и запускает ожидание получения кода пользователем (до 5 минут).

### 3.2. Управление поездками

#### 3.2.1. Описание и приоритет

Блок обеспечивает полный жизненный цикл поездки: создание, редактирование, просмотр. Приоритет: **Must have**.

#### 3.2.2. Причинно-следственные связи

Поездка - центральная сущность приложения. Все остальные сущности контента (документы, расходы, заметки, POI, покупки услуг) привязываются к конкретной поездке. Создание поездки является предусловием для блоков 3.3–3.6.

#### 3.2.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ01 | Пользователь должен иметь возможность создавать поездки, указывая как минимум страну, город и даты путешествия. | Must have |
| ФТ02 | Пользователь должен иметь возможность редактировать страну, город и даты в ранее созданной поездке. | Must have |
| ФТ03 | Пользователь должен иметь возможность просматривать ранее созданные поездки в офлайн-режиме. | Must have |
| ФТ15 | Пользователь должен иметь возможность просматривать поездки, созданные администратором и привязанные к аккаунту пользователя. | Must have |

#### 3.2.4. Алгоритмы

**UC-02-US/Создать свое путешествие (поездку) (Основной поток) **

1.	Система отображает экран создания поездки с полями для ввода параметров поездки (страна, город, даты);
2.	Пользователь указывает параметры в соответствующих полях ввода;
3.	Если пользователь подтверждает создание, нажимая на кнопку “Создать”, то управление переходит на следующий шаг;
4.	Система проверяет корректность введенных параметров;
5.	Если все параметры введены корректно, то управление переходит на следующий шаг;
6.	Если система успешно создает поездку и сохраняет ее с возможностью последующей синхронизации между устройствами, то управление переходит на следующий шаг;
7.	Система открывает экран созданной поездки.

**UC-03-US/Отредактировать параметры поездки (Основной поток)**

1.	Система отображает экран редактирования параметров поездки с текущими значениями;
2.	Пользователь изменяет один или несколько основных параметров поездки (страну, город или даты);
3.	Пользователь подтверждает изменения путем нажатия на кнопку “Сохранить”;
4.	Система проверяет корректность введенных значений;
5.	Если все параметры введены корректно, то управление переходит на следующий шаг;

## 4. Нефункциональные требования

### 4.1. Интерфейсы

#### 4.1.1. Пользовательский интерфейс

-	Приложение должно поддерживать iOS не ниже 14.0 и Android не ниже 9.0 (НФТ01);
-	Русский язык интерфейса: не менее 99% пользовательских строк (исключая пользовательский контент) отображаются на русском языке (НФТ04);
-	Интерфейс должен быть понятен первичному пользователю без обучения: в UX-тесте (≥ 10 пользователей) ≥ 80% успешно выполняют ключевые сценарии без подсказок (НФТ09, приоритет Should have).

#### 4.1.2. Программный интерфейс

REST API серверной части описан в спецификации OpenAPI 3.0.3. API обеспечивает взаимодействие мобильных клиентов с серверным бэкендом. Формат обмена данных: JSON. Аутентификация: Bearer Token (JWT) с механизмом обновления (refresh token).

#### 4.1.3. Интерфейсы передачи данных

-	Взаимодействие клиент-сервер: HTTPS;
-	Хранение файлов: S3-совместимое объектное хранилище;
-	Интеграция с внешними системами (SMS-шлюз, push-сервисы, платёжный агрегатор, OAuth-провайдеры): HTTPS REST API.

### 4.2. Документация для пользователей

В рамках MVP документация для конечных пользователей не предусмотрена.

### 4.3. Лицензионные требования

Конкретные требования к лицензированию на данном этапе не определены. Выбор фреймворка (React Native / Flutter) и используемых библиотек должен учитывать совместимость лицензий с коммерческим использованием.

### 4.4. Предупреждения, касающиеся законодательства, авторских прав и другие замечания

-	Приложение обрабатывает персональные данные пользователей (имя, фамилия, номер телефона, данные социальных сетей) и подлежит требованиям Федерального закона от 27.07.2006 № 152-ФЗ “О персональных данных”;
-	Необходимо обеспечить получение согласия пользователя на обработку персональных данных при регистрации;
-	Функция удаления аккаунта (ФТ10) реализует право субъекта на отзыв согласия и удаление данных;
-	Недопустимы: продажа данных пользователей третьим лицам, размещение прямой рекламы сторонних сервисов.
