## 1. Введение

### 1.1. Назначение

Настоящая спецификация требований к программному обеспечению (SRS) описывает функциональные и нефункциональные требования к выпуску 1.0 (MVP) мобильного приложения Trip plan.

Спецификация предназначена для команд разработки, тестирования, аналитики, а также для заказчика (туристическое агентство) в качестве основы для приёмки результатов работ.

Кроме специально обозначенных случаев, все указанные здесь требования имеют высокий приоритет и приписаны к выпуску 1.0.


### 1.2. Область применения

Trip plan - мобильное приложение-спутник путешественника, разрабатываемое для туристического агентства и решающее ключевые боли:
-	устранение “информационного хаоса” (документы/обсуждения в почте и чатах);
-	доступ к документам и планам в поездке (включая офлайн);
-	возможность агентству предлагать дополнительные услуги в нужный момент;
-	снижение нагрузки на менеджеров по рутинным вопросам.
Полная функциональность перечислена в разделе 3. “Функциональные требования”.


### 1.3. Определения, акронимы и сокращения

1.	**MVP** - минимально жизнеспособная версия продукта.
2.	**OAuth** – протокол авторизации через внешних поставщиков идентификации (VK, Telegram, Яндекс).
3.	**OTP** - одноразовый код подтверждения (SMS).
4.	**POI (Point of Interest)** - точка интереса (место/объект) из каталога.
5.	**Авторизация** – это процесс предоставления определенному пользователю (или группе) прав доступа к ресурсам, функциям или данным системы, а также проверка этих прав при попытке выполнить действие.  
6.	**Администратор (менеджер)** - сотрудник туристического агентства с административными правами.
7.	**Дополнительная услуга** – платная услуга агентства (экскурсия, трансфер, страховка), доступная для приобретения через приложение.
8.	**Офлайн-режим** - доступ к данным без сети, с последующей синхронизацией.
9.	**Поездка (путешествие)** – центральная сущность приложения, описывающая путешествие с указанием страны, городов, дат и связанного контента.
10.	**ПО** – программное обеспечение.
11.	**Пользователь (клиент)** - конечный пользователь приложения, путешественник.
12.	**Приложение/Trip plan** - мобильное приложение (iOS/Android), разработанное в соответствии с данной спецификацией.
13.	**Синхронизация** – процесс передачи изменений, выполненных в офлайн-режиме, на сервер после восстановления сетевого подключения. 
14.	**Участник поездки** – пользователь, добавленный владельцем поездки в качестве соучастника с определенным уровнем доступа (владелец/редактор/наблюдатель).

## 2. Общее описание

### 2.1. Видение продукта

Trip plan решает три ключевые проблемы, обозначенные заказчиком:

1. **Информационный хаос** - документы, планы и обсуждения поездок разбросаны между почтой, мессенджерами и облачными хранилищами. Приложение объединяет всё в едином цифровом пространстве с офлайн-доступом.

2. **Потеря важных файлов в поездке** - пользователь получает надёжное локальное хранилище документов на устройстве с облачной резервной копией.

3. **Невозможность предложить услуги в нужный момент** - при планировании поездки система контекстно предлагает релевантные экскурсии, трансферы и страховки агентства, превращая приложение в канал продаж.

Бизнес-цели MVP: повышение лояльности клиентов (NPS), увеличение среднего чека через кросс-продажи, снижение операционных затрат на рутинные запросы менеджерам.

### 2.2. Перспективы продукта

Функциональность, запланированная на последующие выпуски (за пределами MVP):

- Английская локализация интерфейса.
- Интеграция с Яндекс.Картами / Google Maps API для автоматического построения маршрутов между точками.
- Совместные списки покупок и сборов.
- Голосование за точки маршрута среди участников поездки.
- Интеграция с погодными сервисами.
- Интеграция с CRM агентства.
- Интеграция с API онлайн-киосков билетов (музеи и т. п.).
- Премиум-подписка с расширенной аналитикой и персональными предложениями.
- Адаптивный интерфейс для планшетов.

### 2.3. Взаимодействие продукта с другими продуктами и компонентами

В рамках MVP приложение взаимодействует со следующими внешними системами:

| Внешняя система | Назначение | Протокол |
|---|---|---|
| SMS-шлюз (оператор связи) | Отправка OTP-кодов при регистрации/входе, SMS-уведомления (резервный канал) | REST API |
| Сервис push-уведомлений (APNs / FCM) | Доставка push-уведомлений пользователям | REST API |
| Платёжный агрегатор | Обработка платежей при покупке дополнительных услуг | REST API |
| Поставщики OAuth (VK, Telegram, Яндекс) | Авторизация пользователей через социальные сети | OAuth 2.0 |
| Объектное хранилище (S3-совместимое) | Хранение загруженных документов поездок | S3 API |

### 2.4. Классы и характеристики пользователей

| Класс | Характеристика |
|---|---|
| **Неавторизованный пользователь** | Посетитель, не прошедший авторизацию. Доступны только экраны регистрации и входа. |
| **Пользователь (Клиент)** | Зарегистрированный пользователь приложения (роль `client`). Целевая аудитория: "самостоятельные планировщики" (28–50 лет), с доходом выше среднего, высокой цифровой грамотностью. Доступны: создание и редактирование поездок, управление документами, расходами, заметками, POI, участниками; покупка дополнительных услуг; управление настройками уведомлений; удаление аккаунта. |
| **Администратор (Менеджер)** | Сотрудник туристического агентства (роль `admin`). Доступны: все функции клиента, а также: управление каталогами POI и дополнительных услуг; создание поездок для клиентов; обработка обращений в поддержку и модерация; просмотр аналитических отчётов. |

### 2.5. Ограничения разработки и реализации

- Кроссплатформенная разработка (React Native или Flutter) - приоритетный подход.
- Минимально поддерживаемые версии: iOS 14.0, Android 9.0 (API 28).
- Серверная СУБД: PostgreSQL 16+.
- Формат REST API: OpenAPI 3.0.3, JSON.
- Хранение данных: гибридная модель (облако + устройство).
- Максимальный размер загружаемого файла: 10 МБ. Суммарный объём документов на поездку: 50 МБ.
- Общий лимит облачного хранилища: 1 ГБ на пользователя.
- Поддерживаемые форматы документов: PDF, JPG, PNG.
- Язык интерфейса MVP: русский.
- Недопустимые сценарии монетизации: прямая реклама сторонних сервисов, продажа данных пользователей, навязчивые баннеры.

### 2.6. Допущения и зависимости

- Предполагается, что заказчик предоставляет доступ к SMS-шлюзу, платёжному агрегатору и данные для настройки OAuth-провайдеров до начала интеграционного тестирования.
- Контент каталогов POI и дополнительных услуг наполняется силами заказчика (администраторами) после развёртывания системы.
- Данные справочников стран, городов и валют предзаполнены с ориентацией на российский туристический рынок; расширение справочников выполняется администратором.
- Для офлайн-режима предполагается, что пользователь хотя бы один раз выполнил онлайн-синхронизацию данных поездки.
- Планируемый рост аудитории: 5 000 MAU через 3 месяца, 25 000 через 6 месяцев, 100 000 через 12 месяцев.

## 3. Функциональные требования

Приоритизация требований выполнена по методу MoSCoW. Если не указано иное, требование имеет приоритет **Must have**.

### 3.1. Аутентификация и управление аккаунтом

#### 3.1.1. Описание и приоритет

Блок обеспечивает регистрацию, вход и управление учётной записью пользователя. Приоритет: **Must have**.

#### 3.1.2. Причинно-следственные связи

Аутентификация является входной точкой для всех остальных функциональных блоков. Без успешной авторизации пользователь не получает доступа к функциям приложения. Удаление аккаунта обусловлено требованиями 152-ФЗ.

#### 3.1.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ18 | Система должна поддерживать регистрацию и вход пользователя с использованием номера телефона и одноразового кода подтверждения (OTP). | Must have |
| ФТ20 | Система должна обеспечивать отправку одноразового кода подтверждения на номер телефона пользователя и проверку кода при входе. Срок действия кода - 5 минут. | Must have |
| ФТ22 | Система должна поддерживать регистрацию и вход пользователя через внешних поставщиков идентификации: VK, Telegram, Яндекс (в указанном порядке приоритета). | Must have |
| ФТ23 | Система должна поддерживать роли Пользователь (`client`) и Администратор (`admin`). | Must have |
| ФТ10 | Пользователь должен иметь возможность удалить аккаунт с полным удалением персональных данных в соответствии с 152-ФЗ. Удаление доступно только в онлайн-режиме. | Must have |

#### 3.1.4. Алгоритмы

**Регистрация / вход по номеру телефона (UC-01-US)**

1. Пользователь вводит номер телефона.
2. Система отправляет OTP-код через SMS-шлюз (UC-03-ITG).
3. Пользователь вводит полученный код в течение 5 минут.
4. Система верифицирует код.
5. Если номер не зарегистрирован - система запрашивает имя и фамилию, отображает согласие на обработку персональных данных, создаёт учётную запись.
6. Если номер уже зарегистрирован - система авторизует пользователя и переходит на главный экран.
7. При неполучении кода пользователь может запросить повторную отправку. При повторной неудаче - возврат на стартовый экран.
8. При вводе неверного кода - алерт об ошибке, возврат на стартовый экран.

**Регистрация / вход через социальные сети**

1. Пользователь выбирает провайдера (VK, Telegram, Яндекс).
2. Система перенаправляет на OAuth-авторизацию провайдера.
3. Пользователь подтверждает доступ.
4. Система получает данные профиля, создаёт или привязывает учётную запись.
5. Переход на главный экран приложения.

**Удаление аккаунта (UC-06-US)**

1. Пользователь инициирует удаление в настройках профиля.
2. Система отображает предупреждение и запрашивает подтверждение.
3. При подтверждении - система выполняет soft-delete учётной записи (is_active = false), удаляет персональные данные.
4. Сессия завершается, отображается стартовый экран.
5. При отсутствии сети - отображается сообщение о необходимости подключения.

### 3.2. Управление поездками

#### 3.2.1. Описание и приоритет

Блок обеспечивает полный жизненный цикл поездки: создание, редактирование, просмотр. Приоритет: **Must have**.

#### 3.2.2. Причинно-следственные связи

Поездка - центральная сущность приложения. Все остальные сущности контента (документы, расходы, заметки, POI, покупки услуг) привязываются к конкретной поездке. Создание поездки является предусловием для блоков 3.3–3.6.

#### 3.2.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ01 | Пользователь должен иметь возможность создавать поездки, указывая как минимум страну, город и даты путешествия. | Must have |
| ФТ02 | Пользователь должен иметь возможность редактировать страну, город и даты в ранее созданной поездке. | Must have |
| ФТ03 | Пользователь должен иметь возможность просматривать ранее созданные поездки в офлайн-режиме. | Must have |
| ФТ15 | Пользователь должен иметь возможность просматривать поездки, созданные администратором и привязанные к аккаунту пользователя. | Must have |
| ФТ19 | При создании или изменении направления поездки система должна предлагать пользователю релевантные точки интереса и дополнительные услуги, соответствующие выбранному направлению. | Must have |

#### 3.2.4. Алгоритмы

**Создание поездки (UC-02-US)**

1. Пользователь выбирает действие "Создать поездку".
2. Система отображает форму с полями: страна, город (из справочников), даты начала и окончания.
3. Пользователь заполняет параметры и нажимает "Создать".
4. Система проверяет корректность (обязательные поля заполнены, дата окончания ≥ даты начала).
5. При успешной валидации - поездка создаётся и сохраняется с возможностью синхронизации.
6. Система отображает экран созданной поездки с предложением добавить участников, документы, POI и расходы.
7. При ошибке валидации - отображается сообщение с указанием полей для исправления.
8. При ошибке сохранения - алерт с предложением повторить.

**Редактирование поездки (UC-03-US)**

1. Пользователь открывает экран редактирования поездки.
2. Система отображает текущие значения параметров.
3. Пользователь изменяет параметры и нажимает "Сохранить".
4. Система валидирует, сохраняет и синхронизирует изменения.
5. Последовательно предлагаются действия: добавить участников → загрузить документы → добавить POI → добавить расходы (каждое можно пропустить).

**Просмотр поездки, созданной менеджером (UC-04-US)**

1. Пользователь переходит к списку поездок.
2. Система отображает доступные поездки, включая созданные администратором.
3. При онлайн-режиме данные загружаются с сервера; при офлайн-режиме - из локального хранилища (при наличии).
4. Если локальных данных нет и сеть отсутствует - алерт с предложением повторить при появлении сети.

### 3.3. Контент поездки: документы, расходы, заметки

#### 3.3.1. Описание и приоритет

Блок обеспечивает работу с документами, расходами и заметками в рамках поездки, включая офлайн-режим. Приоритет: **Must have**.

#### 3.3.2. Причинно-следственные связи

Функции данного блока доступны только при наличии созданной поездки (блок 3.2). Расходы агрегируются для формирования сводки и отчёта (ФТ07, ФТ16). Документы хранятся в объектном хранилище и доступны офлайн при предварительной загрузке.

#### 3.3.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ04 | Пользователь должен иметь возможность вносить расходы, указав сумму, категорию, валюту и дату, в офлайн-режиме. | Must have |
| ФТ05 | Пользователь должен иметь возможность вносить заметки в поездку в офлайн-режиме. | Must have |
| ФТ06 | Пользователь должен иметь возможность прикреплять к поездке файлы документов. | Must have |
| ФТ07 | Пользователь должен иметь возможность просматривать суммарные расходы по поездке и по категориям. | Must have |
| ФТ28 | Система должна ограничивать размер загружаемого документа значением не более 10 МБ. | Must have |
| ФТ29 | Система должна позволять добавлять документы как из интерфейса приложения, так и путём импорта через функцию "Поделиться" из других приложений. | Should have |
| ФТ16 | Пользователь должен иметь возможность экспортировать отчёт по расходам в форматах PDF и CSV. | Should have |

#### 3.3.4. Алгоритмы

**Загрузка документов (UC-01-EX)**

Два сценария загрузки:

*Через функцию "Поделиться":*
1. Пользователь выбирает файл в стороннем приложении и нажимает "Поделиться" → Trip plan.
2. Система принимает файл и предлагает выбрать поездку для привязки.
3. Система проверяет формат (PDF, JPG, PNG) и размер (≤ 10 МБ).
4. При успешной проверке - документ сохраняется и привязывается к поездке.

*Через интерфейс приложения:*
1. Пользователь на экране поездки нажимает "Загрузить документ".
2. Система открывает системный выбор файла.
3. Далее - аналогично шагам 3–4 выше.

При неподдерживаемом формате - сообщение об ошибке с предложением выбрать другой файл. При ошибке загрузки - предложение повторить.

**Управление расходами (UC-02-EX)**

1. Пользователь открывает раздел "Расходы" в поездке.
2. Система отображает список расходов и действие "Добавить расход".
3. Пользователь заполняет форму: сумма (> 0), категория (из справочника, включая пользовательские), валюта, дата, описание (опционально).
4. Система валидирует и сохраняет расход.
5. В офлайн-режиме расход сохраняется локально с флагом `is_synced = false` и синхронизируется при появлении сети.
6. Система пересчитывает и отображает обновлённый список.

Предустановленные категории расходов: Перелёт, Жильё, Питание, Транспорт, Развлечения, Шоппинг, Сувениры. Пользователь может создавать собственные категории.

### 3.4. Точки интереса (POI)

#### 3.4.1. Описание и приоритет

Блок обеспечивает просмотр курируемого каталога POI агентства и добавление точек интереса в план поездки. Приоритет: **Must have**.

#### 3.4.2. Причинно-следственные связи

Каталог POI наполняется администраторами (блок 3.8). Пользователь добавляет POI из каталога в поездку; при изменении направления поездки система предлагает релевантные POI (ФТ19). POI отображаются как в каталоге, так и на интерактивной карте.

#### 3.4.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ08 | Пользователь должен иметь возможность добавлять точки интереса (POI) из каталога в план поездки. | Must have |
| ФТ09 | Пользователь должен иметь возможность выбирать точки интереса на интерактивной карте. | Must have |

#### 3.4.4. Алгоритмы

**Добавление POI в поездку (UC-01-EX "Добавить в поездку точки интереса")**

*Через каталог:*
1. Система отображает подборки POI агентства для выбранного направления (страна/город поездки).
2. Пользователь открывает подборку и выбирает POI.
3. Система отображает карточку POI (название, описание, адрес, координаты, категория).
4. Пользователь нажимает "Добавить в поездку".
5. Система добавляет POI в план поездки, сохраняет и синхронизирует изменения.

*Через интерактивную карту:*
1. Система отображает карту с отметками POI для выбранного направления.
2. Пользователь выбирает точку на карте.
3. Далее - аналогично шагам 3–5 каталога.

При отмене - возврат к списку без сохранения. При ошибке синхронизации - алерт с предложением повторить.

### 3.5. Совместные поездки

#### 3.5.1. Описание и приоритет

Блок обеспечивает многопользовательский режим работы с поездками. Приоритет: **Must have**.

#### 3.5.2. Причинно-следственные связи

Совместные поездки являются ключевой функцией для семейного и группового планирования. Владелец поездки управляет составом участников и правами доступа. Участники получают доступ к общему плану и сводке расходов.

#### 3.5.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ21 | Система должна обеспечивать возможность добавления других пользователей в качестве участников поездки. | Must have |
| ФТ25 | При совместной поездке система должна предоставлять всем участникам доступ к просмотру общего плана и сумме расходов по поездке. | Must have |
| ФТ27 | При добавлении участников поездки система должна позволять владельцу управлять правами доступа участников. | Must have |

#### 3.5.4. Алгоритмы

**Добавление участников (UC-01-EX "Добавить участников поездки")**

1. Владелец поездки нажимает "Добавить участников".
2. Система отображает экран добавления участников с настройкой прав доступа.
3. Владелец вводит данные участника и настраивает роль: `editor` (редактирование) или `viewer` (только просмотр).
4. Система проверяет корректность данных.
5. При успешной валидации - участник добавлен, изменения сохранены и синхронизированы.
6. Участнику отправляется уведомление (при включённых настройках).

Роли участников: `owner` (создатель, полный доступ), `editor` (просмотр и редактирование), `viewer` (только просмотр).

### 3.6. Дополнительные услуги и платежи

#### 3.6.1. Описание и приоритет

Блок обеспечивает приобретение дополнительных услуг агентства через приложение. Приоритет: **Must have**.

#### 3.6.2. Причинно-следственные связи

Каталог услуг наполняется администраторами (блок 3.8). Покупка доступна только в онлайн-режиме и требует интеграции с платёжным агрегатором. Покупка привязывается к поездке и пользователю. Данные о покупках используются для формирования отчётов по конверсии (блок 3.8).

#### 3.6.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ14 | Пользователь должен иметь возможность приобретать дополнительные услуги, доступные для выбранной поездки, с фиксацией результата операции. | Must have |

#### 3.6.4. Алгоритмы

**Покупка дополнительной услуги (UC-05-US)**

1. Система отображает каталог дополнительных услуг, доступных для поездки (экскурсии, трансферы, страховки).
2. Пользователь выбирает услугу.
3. Система отображает информацию об услуге (описание, цена, валюта) и предлагает подтвердить покупку.
4. Пользователь нажимает "Купить".
5. Система проверяет наличие сети (покупка только онлайн).
6. Система инициирует процесс оплаты через платёжный агрегатор.
7. Агрегатор обрабатывает платёж и возвращает результат.
8. При успешном платеже - покупка фиксируется (статус `completed`), пользователю отображается подтверждение.
9. При отклонении платежа - алерт об ошибке оплаты с предложением повторить.
10. При отсутствии сети - сообщение о невозможности покупки без интернета.

Статусы платежа: `pending` → `completed` / `failed` / `refunded`.

### 3.7. Уведомления

#### 3.7.1. Описание и приоритет

Блок обеспечивает отправку сервисных и маркетинговых уведомлений пользователям. Приоритет: **Must have** (сервисные), **Should have** (управление настройками).

#### 3.7.2. Причинно-следственные связи

Уведомления инициируются событиями: приближение даты поездки, изменения в данных совместной поездки, результат обработки обращения. Доставка зависит от интеграций с push-сервисами и SMS-шлюзом, а также от пользовательских настроек.

#### 3.7.3. Функциональные требования

| ID | Требование | Приоритет |
|---|---|---|
| ФТ24 | Система должна отправлять сервисные уведомления о событиях поездки и/или изменениях данных поездки при включённых настройках уведомлений. | Must have |
| ФТ17 | Пользователь должен иметь возможность управлять настройками уведомлений (push, SMS, маркетинговые). | Should have |

#### 3.7.4. Алгоритмы

**Отправка напоминания о поездке (UC-01-ITG)**

1. Система определяет наступление события (например, за день до вылета).
2. Система проверяет, разрешены ли сервисные уведомления для пользователя.
3. Если разрешены - формируются параметры уведомления, выбирается канал (push - приоритетный).
4. Если push-канал доступен - отправка через push-сервис.
5. Если push недоступен, но доступен SMS - отправка через SMS-шлюз.
6. Если оба канала недоступны - завершение без фиксации успешной отправки.
7. При успешной отправке - фиксация факта доставки.

**Уведомление об изменениях (UC-02-ITG)** - аналогичный алгоритм, триггер: изменение данных поездки (например, другим участником).

Типы уведомлений: `reminder` (напоминания о поездке), `change` (изменения данных), `marketing` (маркетинговые, только при подписке).

Пользовательские настройки: `notifications_push`, `notifications_sms`, `notifications_marketing` - каждая настройка управляется независимо.

**Управление каталогом POI (UC-04-AD)**

1. Администратор открывает раздел "Каталог POI".

3. *Создание/Редактирование:* администратор заполняет/изменяет поля (название, описание, координаты, адрес, категория, город); система валидирует и сохраняет.
4. *Удаление:* подтверждение → удаление из каталога.
5. При ошибке сохранения - алерт с предложением повторить.

**Настройка каталога дополнительных услуг (UC-05-AD)**

Алгоритм аналогичен управлению каталогом POI. Поля услуги: название, описание, цена, валюта, тип услуги (`excursion`, `transfer`, `insurance`, `other`), привязка к городу (опционально).

**Создание поездки для клиента (UC-03-AD)**

1. Администратор выбирает "Создать поездку для клиента".
2. Система предлагает выбрать клиента из списка пользователей.
3. Администратор заполняет параметры поездки (страна, город, даты).
4. Система валидирует, создаёт поездку и привязывает к аккаунту клиента.
5. Клиент получает доступ к поездке в своём списке.

**Обработка обращений (UC-02-AD)**

1. Администратор выбирает режим: "Обращения в поддержку" или "Модерация".
2. Система отображает список обращений/объектов модерации.
3. Администратор выбирает обращение, меняет статус на "В работе", обрабатывает.
4. При решении - статус "Решено"; при невозможности решения - "Требует доработки" с комментарием.
5. Статусы обращений: `open` → `in_progress` → `resolved` / `needs_work`.

**Формирование отчётов**

Виды отчётов:
- *Популярность направлений:* период (месяц/квартал), фильтр по сегментам, данные: ТОП-10 стран/городов.
- *Конверсия в услуги:* период (месяц), данные: доля пользователей, добавивших POI и купивших экскурсию.
- *Активность пользователей:* произвольный период, данные: DAU/WAU, среднее количество действий.

## 4. Нефункциональные требования

### 4.1. Интерфейсы

#### 4.1.1. Пользовательский интерфейс

- Приложение должно поддерживать iOS не ниже 14.0 и Android не ниже 9.0 (НФТ01).
- Русский язык интерфейса: не менее 99% пользовательских строк (исключая пользовательский контент) отображаются на русском языке (НФТ04).

#### 4.1.2. Программный интерфейс

REST API серверной части описан в спецификации OpenAPI 3.0.3 (файл `openapi.yaml`, см. Приложение А). API обеспечивает взаимодействие мобильных клиентов с серверным бэкендом. Формат обмена данных: JSON. Аутентификация: Bearer Token (JWT) с механизмом обновления (refresh token).

#### 4.1.3. Интерфейсы передачи данных

- Взаимодействие клиент-сервер: HTTPS (TLS 1.2+).
- Хранение файлов: S3-совместимое объектное хранилище.
- Интеграция с внешними системами (SMS-шлюз, push-сервисы, платёжный агрегатор, OAuth-провайдеры): HTTPS REST API.

- Разграничение доступа по ролям (`client`, `admin`) и правам участника поездки (`owner`, `editor`, `viewer`).
- Soft-delete при удалении аккаунта для обеспечения аудита и соблюдения 152-ФЗ.
