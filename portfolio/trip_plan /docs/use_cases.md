# Диаграмма вариантов использования Trip plan
 
> ![Диаграмма вариантов использования](diagrams/use-case-diagram.png)
 
## I. Описание основных вариантов использования
 
### 1. Текстовое описание варианта использования «Зарегистрироваться/войти в приложение»
 
| Поле | Описание |
|---|---|
| **ID/Use Case** | UC-01-US/Зарегистрироваться/войти в приложение |
| **Краткое описание** | Данный use case описывает процесс регистрации нового пользователя в системе для получения доступа к личному кабинету. |
| **Действующие лица** | Пользователь (клиент) |
| **Триггер** | Пользователь решает зарегистрироваться и нажимает на кнопку «Регистрация». |
| **Предусловия** | Пользователь не авторизован и находится на экране регистрации приложения. |
| **Основной поток** | 1. *Если* пользователь выбирает регистрацию по номеру телефона, то управление переходит на следующий шаг;<br>2. Система выводит экран для ввода номера телефона;<br>3. Пользователь вводит номер телефона;<br>4. Пользователь нажимает на кнопку «Далее»;<br>5. *Если* телефон пользователя не был зарегистрирован ранее, то управление переходит на следующий шаг;<br>6. Система отправляет пользователю одноразовый код для входа в систему на номер телефона;<br>7. Система выводит экран с полем ввода кода из СМС;<br>8. *Если* пользователь получил одноразовый код в течение 5 минут, то управление переходит на следующий шаг;<br>9. Пользователь вводит код из СМС;<br>10. Система верифицирует введенные данные;<br>11. *Если* введенный и отправленный ранее код совпадают, то управление переходит на следующий шаг;<br>12. Система выводит экран с полями ввода имени и фамилии пользователя;<br>13. Пользователь вводит свои имя и фамилию в форму;<br>14. Пользователь нажимает на кнопку «Далее»;<br>15. Система выводит алерт о согласии на обработку персональных данных;<br>16. Пользователь нажимает на кнопку «Согласен с условиями обработки»;<br>17. Система создает учетную запись для пользователя;<br>18. Система выводит главную страницу приложения. |
| **Альтернативный поток** | **1а.** Пользователь выбирает регистрацию через социальную сеть<br>1. Система выводит экран с перечнем социальных сетей для регистрации (VK, Telegram, Яндекс);<br>2. Пользователь выбирает один из вариантов;<br>3. Система предлагает пользователю выбрать аккаунт для регистрации;<br>4. Пользователь выбирает аккаунт;<br>5. Пользователь нажимает кнопку «Далее»;<br>6. Управление переходит на шаг 14.<br><br>**5а.** Телефон пользователя уже был зарегистрирован ранее<br>1. Система отправляет пользователю одноразовый код для входа в систему на номер телефона;<br>2. Система выводит экран с полем ввода кода из СМС;<br>3. Пользователь вводит код из СМС;<br>4. Система верифицирует введенные данные;<br>5. Управление переходит на шаг 18.<br><br>**8а.** Пользователь не получил код в течение 5 минут<br>1. Пользователь нажимает на кнопку «Отправить код еще раз»;<br>2. Система повторно отправляет пользователю новый одноразовый код на номер телефона;<br>3. Управление переходит на шаг 9. |
| **Поток исключения** | **8b.** Пользователь не получил повторный одноразовый код<br>1. Пользователь нажимает кнопку «Я не получил код»;<br>2. Система выводит стартовый экран регистрации приложения;<br>3. Вариант использования завершает свою работу.<br><br>**11а.** Пользователь ввел неверный код из СМС<br>1. Система выводит алерт об ошибке ввода;<br>2. Система выводит стартовый экран регистрации приложения;<br>3. Вариант использования завершает свою работу. |
| **Постусловие** | В случае успешного выполнения основного потока, у пользователя появляется доступ в личный кабинет в приложении. |
| **Результат** | Пользователь зарегистрирован в приложении. |
 
### 2. Текстовое описание варианта использования «Создать свое путешествие (поездку)»
 
| Поле | Описание |
|---|---|
| **ID/Use Case** | UC-02-US/Создать свое путешествие (поездку) |
| **Краткое описание** | Данный use case описывает процесс создания пользователем нового путешествия (поездки) в приложении Trip plan. |
| **Действующие лица** | Пользователь (клиент) |
| **Триггер** | Пользователь решает создать новую поездку и выбирает действие «Создать поездку». |
| **Предусловия** | Пользователь авторизован в приложении и находится на главном экране приложения. |
| **Основной поток** | 1. Система отображает экран создания поездки с полями для ввода параметров поездки (страна, город, даты);<br>2. Пользователь указывает параметры в соответствующих полях ввода;<br>3. *Если* пользователь подтверждает создание, нажимая на кнопку «Создать», то управление переходит на следующий шаг;<br>4. Система проверяет корректность введенных параметров;<br>5. *Если* все параметры введены корректно, то управление переходит на следующий шаг;<br>6. *Если* система успешно создает поездку и сохраняет ее с возможностью последующей синхронизации между устройствами, то управление переходит на следующий шаг;<br>7. Система открывает экран созданной поездки. |
| **Альтернативный поток** | **5а.** Пользователь не заполнил обязательные параметры/заполнил их некорректно<br>1. Система отображает сообщение о необходимости заполнить/исправить обязательные поля;<br>2. Пользователь вносит недостающие данные/изменяет их;<br>3. Система повторно проверяет корректность параметров;<br>4. Управление переходит на шаг 6. |
| **Поток исключения** | **3а.** Пользователь отменяет создание поездки<br>1. Пользователь на экране создания поездки нажимает на кнопку «Отмена»;<br>2. Система выводит главный экран приложения, поездка не создается;<br>3. Вариант использования завершает свою работу.<br><br>**6а.** Ошибка создания или сохранения поездки<br>1. Система отображает пользователю алерт об ошибке и предлагает повторить процесс создания поездки;<br>2. Пользователь нажимает на кнопку «Закрыть»;<br>3. Система выводит главный экран приложения, поездка не создается;<br>4. Вариант использования завершает свою работу. |
| **Постусловие** | В случае успешного выполнения основного потока, в системе появляется новая поездка, доступная пользователю в списке его путешествий. |
| **Результат** | Пользователь создал свое путешествие (поездку) в приложении. |
 
### 3. Текстовое описание варианта использования «Отредактировать параметры поездки»
 
| Поле | Описание |
|---|---|
| **ID/Use Case** | UC-03-US/Отредактировать параметры поездки |
| **Краткое описание** | Данный use case описывает процесс редактирования пользователем параметров ранее созданной поездки, чтобы актуализировать план поездки в приложении. |
| **Действующие лица** | Пользователь (клиент) |
| **Триггер** | Пользователь на экране поездки выбирает действие «Отредактировать параметры поездки». |
| **Предусловия** | Поездка уже создана авторизированным пользователем, который находится на странице поездки. |
| **Основной поток** | 1. Система отображает экран редактирования параметров поездки с текущими значениями;<br>2. Пользователь изменяет один или несколько основных параметров поездки (страну, город или даты);<br>3. Пользователь подтверждает изменения путем нажатия на кнопку «Сохранить»;<br>4. Система проверяет корректность введенных значений;<br>5. *Если* все параметры введены корректно, то управление переходит на следующий шаг;<br>6. Система выводит экран, на котором предлагает пользователю добавить других участников в поездку;<br>7. *Если* пользователь нажимает на кнопку «Пропустить», то управление переходит на следующий шаг;<br>8. Система выводит экран, на котором предлагает пользователю загрузить документы по поездке;<br>9. *Если* пользователь нажимает на кнопку «Пропустить», то управление переходит на следующий шаг;<br>10. Система выводит экран, на котором предлагает пользователю добавить в поездку точки интереса (POI) из подборок агентства;<br>11. *Если* пользователь нажимает на кнопку «Пропустить», то управление переходит на следующий шаг;<br>12. Система выводит экран, на котором предлагает пользователю добавить в поездку запланированные/понесенные расходы по категориям;<br>13. *Если* пользователь нажимает на кнопку «Пропустить», то управление переходит на следующий шаг;<br>14. Система сохраняет обновленные параметры поездки и синхронизирует данные;<br>15. Система отображает обновленные параметры на экране поездки. |
| **Альтернативный поток** | **5а.** Пользователь не заполнил обязательные параметры/заполнил их некорректно<br>1. Система отображает сообщение о необходимости заполнить/исправить обязательные поля;<br>2. Пользователь вносит недостающие данные/изменяет их;<br>3. Система повторно проверяет корректность параметров;<br>4. Управление переходит на шаг 6.<br><br>**7а.** Пользователь решает добавить других участников в поездку<br>1. Система запускает выполнение use case «Добавить участников поездки»;<br>2. Расширяющий вариант использования «Добавить участников поездки» завершается;<br>3. Система возвращает пользователя к редактированию параметров поездки;<br>4. Управление переходит на шаг 8.<br><br>**9а.** Пользователь решает загрузить документы по поездке<br>1. Система запускает выполнение use case «Загрузить документы по поездке»;<br>2. Расширяющий вариант использования «Загрузить документы по поездке» завершается;<br>3. Система возвращает пользователя к редактированию параметров поездки;<br>4. Управление переходит на шаг 10.<br><br>**11а.** Пользователь решает добавить в поездку точки интереса (POI)<br>1. Система запускает выполнение use case «Добавить в поездку точки интереса (POI)»;<br>2. Расширяющий вариант использования «Добавить в поездку точки интереса (POI)» завершается;<br>3. Система возвращает пользователя к редактированию параметров поездки;<br>4. Управление переходит на шаг 12.<br><br>**13а.** Пользователь решает добавить в поездку расходы по категориям<br>1. Система запускает выполнение use case «Управлять расходами»;<br>2. Расширяющий вариант использования «Управлять расходами» завершается;<br>3. Система возвращает пользователя к редактированию параметров поездки;<br>4. Управление переходит на шаг 14. |
| **Поток исключения** | **3а.** Пользователь отменяет редактирование поездки<br>1. Пользователь на экране редактирования поездки нажимает на кнопку «Отмена»;<br>2. Система выводит главный экран приложения, поездка не изменяется;<br>3. Вариант использования завершает свою работу.<br><br>**14а.** Ошибка сохранения изменений в поездке<br>1. Система отображает пользователю алерт об ошибке и предлагает повторить процесс редактирования поездки;<br>2. Пользователь нажимает на кнопку «Закрыть»;<br>3. Система выводит главный экран приложения, параметры поездки не изменяются;<br>4. Вариант использования завершает свою работу. |
| **Постусловие** | В случае успеха параметры поездки обновлены, сохранены и отображаются пользователю на экране поездки. |
| **Результат** | Параметры поездки отредактированы пользователем. |
 
### 4. Текстовое описание варианта использования «Просмотреть поездку, созданную менеджером»
 
| Поле | Описание |
|---|---|
| **ID/Use Case** | UC-04-US/Просмотреть поездку, созданную менеджером |
| **Краткое описание** | Данный use case описывает процесс просмотра пользователем поездки, которая была заранее создана менеджером агентства и доступна пользователю в приложении. |
| **Действующие лица** | Пользователь (клиент) |
| **Триггер** | Пользователь решает посмотреть поездку и переходит на экран карточек всех своих поездок. |
| **Предусловия** | — Пользователь авторизован в приложении;<br>— В системе существует поездка, созданная менеджером и привязанная к аккаунту пользователя;<br>— Для просмотра офлайн: данные поездки/документы доступны на устройстве пользователя. |
| **Основной поток** | 1. *Если* созданная менеджером поездка доступна для пользователя, то управление переходит на следующий шаг;<br>2. Система показывает пользователю возможность перейти к просмотру поездки;<br>3. Пользователь выбирает поездку, созданную менеджером;<br>4. *Если* просмотр осуществляется в онлайн режиме, то управление переходит на следующий шаг;<br>5. Система загружает данные поездки;<br>6. Система открывает экран поездки и отображает данные поездки. |
| **Альтернативный поток** | **4а.** Просмотр осуществляется в офлайн режиме<br>1. Система проверяет наличие данных поездки, доступных локально на устройстве;<br>2. *Если* локальные данные есть, то управление переходит на следующий шаг;<br>3. Система загружает последнюю доступную (скачанную) версию данных;<br>4. Управление переходит на шаг 6. |
| **Поток исключения** | **1а.** Поездка недоступна для пользователя<br>1. Система информирует пользователя через алерт, что поездка, созданная менеджером, не найдена/недоступна;<br>2. Пользователь выбирает действие «Закрыть»;<br>3. Вариант использования завершает свою работу.<br><br>**4b.** Локальных данных нет<br>1. Система показывает алерт об ошибке;<br>2. Система предлагает повторить попытку при появлении соединения и завершает просмотр;<br>3. Вариант использования завершает свою работу. |
| **Постусловие** | Пользователю отображен экран поездки, созданной менеджером. |
| **Результат** | Пользователь просмотрел поездку, созданную менеджером. |
 
### 5. Текстовое описание варианта использования «Приобрести дополнительные услуги»
 
| Поле | Описание |
|---|---|
| **ID/Use Case** | UC-05-US/Приобрести дополнительные услуги |
| **Краткое описание** | Данный use case описывает процесс приобретения пользователем дополнительных услуг туристического агентства через приложение Trip plan. |
| **Действующие лица** | Пользователь (клиент), Платежный агрегатор |
| **Триггер** | Пользователь желает приобрести дополнительную(-ые) услуги к поездке и открывает соответствующий раздел приложения. |
| **Предусловия** | — Пользователь авторизован в приложении;<br>— У пользователя есть поездка, для которой доступна покупка дополнительных услуг;<br>— В системе доступны каталоги дополнительных услуг агентства;<br>— Для выполнения покупки доступен онлайн-режим и платежный шлюз. |
| **Основной поток** | 1. Система отображает пользователю каталог доступных дополнительных услуг для покупки;<br>2. *Если* пользователь выбирает одну из дополнительных услуг, то управление переходит на следующий шаг;<br>3. Система отображает информацию о выбранной услуге и предлагает подтвердить покупку;<br>4. *Если* пользователь нажимает на кнопку «Купить», то управление переходит на следующий шаг;<br>5. *Если* покупка осуществляется в онлайн режиме, то управление переходит на следующий шаг;<br>6. Система инициирует процесс оплаты через платежный шлюз;<br>7. Платежный агрегатор обрабатывает платеж и возвращает системе результат операции;<br>8. *Если* платеж завершен успешно, то управление переходит на следующий шаг;<br>9. Система фиксирует покупку услуги за пользователем и отображает подтверждение успешной покупки. |
| **Альтернативный поток** | **4а.** Пользователь передумал покупать выбранную ранее услугу<br>1. Пользователь закрывает карточку выбранной услуги;<br>2. Система возвращает пользователя в каталог;<br>3. Пользователь выбирает другую услугу;<br>4. Система отображает информацию о данной услуге и предлагает подтвердить покупку;<br>5. Пользователь нажимает на кнопку «Купить»;<br>6. Управление переходит на шаг 5. |
| **Поток исключения** | **2а.** Пользователь не выбрал услугу для покупки<br>1. Пользователь закрывает каталог дополнительных услуг;<br>2. Вариант использования завершает свою работу.<br><br>**5а.** Отсутствует интернет-соединение<br>1. Система отображает сообщение о невозможности выполнить покупку без доступа к сети;<br>2. Система предлагает повторить попытку при появлении соединения;<br>3. Вариант использования завершает свою работу.<br><br>**8а.** Платеж отклонен платежным агрегатором<br>1. Система отображает пользователю алерт об ошибке оплаты;<br>2. Система предлагает повторить попытку оплаты;<br>3. Пользователь закрывает алерт;<br>4. Вариант использования завершает свою работу. |
| **Постусловие** | В случае успешного выполнения основного потока дополнительная услуга приобретена пользователем, и покупка зафиксирована в системе. |
| **Результат** | Пользователь приобрел дополнительную услугу через приложение Trip plan. |
 
### 6. Текстовое описание варианта использования «Удалить аккаунт»
 
| Поле | Описание |
|---|---|

| **Действующие лица** | Пользователь (клиент) |

| **Предусловия** | — Поездка создана и доступна пользователю;<br>— Пользователь авторизован в приложении;<br>— В системе доступны курируемые подборки POI для выбранного направления (страна/город поездки). |
| **Основной поток** | 1. *Если* пользователь выбирает просмотр точек интереса через каталог, то управление переходит на следующий шаг;<br>2. Система отображает экран добавления POI в поездку и показывает подборки POI агентства для выбранного направления;<br>3. Пользователь открывает подборку и выбирает POI для добавления в поездку;<br>4.
